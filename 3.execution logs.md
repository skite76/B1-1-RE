
# LLM 모델별 회의록 요약 성능 비교 실행 로그

## 1. 과업 개요

### 과업 시나리오
제1회 「행복마을 이웃 사촌 축제」 실행 계획 회의 원문을 바탕으로, 3개 LLM 모델이 회의록 요약, 수치 검증, 담당업무 추출, 주민 안내문 작성, 조건부 재판단까지 수행하는지 비교했습니다.

### 비교 대상 모델
- Claude Opus 4.8
- Gemini 3.1 Pro
- GPT-5.5

### 주요 요청 흐름
1. 회의 원문 요약
2. 이벤트 기획자 관점의 주민 설득용 요약
3. 제시된 수치의 정확성 검증
4. 원문에 없는 수치 확인
5. 원문 수치만 사용한 재요약
6. 전체 예산 1억 원 가정 시 렌탈 비용 계산
7. 결정사항과 Action Items 추출
8. 담당자별 제안내용·업무 또는 행사내용별 문제점·대안 정리
9. 친절하고 자상한 주민 안내문 작성
10. 전력 충분, 렌탈비 10% 이내 조건에서 에어바운스 재검토
11. 최종 주민 제공용 회의록 요약본 작성

# 2. 모델별 실행 결과 요약

## A. Claude Opus 4.8

### 잘한 점
- 전체 회의 흐름을 안정적으로 파악했습니다.
- 주요 결정사항을 정확히 정리했습니다.
  - 에어바운스 대신 야간 등불 축제
  - 푸드트럭 대신 배달 존
  - 쓰레기 되가져가기 캠페인
  - 우천 시 지하 주차장 1구역 활용
- 원문에 있는 수치와 없는 수치를 비교적 잘 구분했습니다.
- 주민 안내문 작성 시 부드럽고 친절한 톤을 잘 유지했습니다.
- “전체 예산 1억 원일 때 렌탈 비용 4,000만 원” 계산도 정확했습니다.

### 문제 발생 지점
1. **확실한 업무 일부 누락**
   - Action Items 정리에서 박현우의 “금요일까지 광장 전력 점검”만 포함했습니다.
   - 그러나 원문에는 다음 업무도 명확히 있습니다.
     - 이지아: 월요일까지 등불 축제 포스터 시안 제작
     - 김민수: 상가 번영회와 배달 존 협의 마무리
     - 이지아: 주민 참여 메시지 카드 제작
     - 이지아: 실내 동선 준비

2. **확실한 문제점을 ‘확인 필요’로 과분류**
   - 먹거리 문제점, 쓰레기 문제점, 우천 대비 문제점을 일부 “확인 필요”로 처리했습니다.
   - 하지만 원문에는 근거가 있습니다.
     - 먹거리: 상권 침해 문제로 항의가 들어옴
     - 쓰레기: 배달 용기가 많을 것이라는 우려
     - 우천 대비: 야외 행사라 날씨가 관건, 주차 공간 부족 우려

3. **사용자 가정과 원문 사실의 구분 필요**
   - “전력 충분 + 렌탈비 10% 이내”는 원문 내용이 아니라 사용자가 제시한 가정입니다.
   - Claude는 대체로 잘 반영했지만, 최종본에서는 이 조건이 사용자 가정임을 더 명확히 표시할 필요가 있었습니다.

### 수정 결과
- 원문에 없는 총예산, 목표 인원, 부스 개수 등은 제거했습니다.
- 예산의 40%, 지하 주차장 1구역, 11월 2일 오후 2시 등 원문 근거 수치만 사용했습니다.
- 최종본은 주민 안내문으로 활용하기에 표현이 안정적이었습니다.
- 다만 확실한 업무와 문제점을 빠뜨리지 않도록 보완이 필요합니다.

## B. Gemini 3.1

### 잘한 점
- 주민 안내문처럼 자연스럽고 설득력 있는 문장을 잘 작성했습니다.
- 친절하고 자상한 톤 전환이 우수했습니다.
- 회의 흐름을 읽기 쉽게 구조화했습니다.
- 사용자의 수치 검증 요청 이후, 원문에 없는 수치를 인정하고 수정했습니다.

### 문제 발생 지점
1. **원문에 없는 수치 생성**
   - 다음 표현은 원문에 없습니다.
     - 상권 침해 0%
     - 상생 100%
     - 수거율 100%
     - 모든 주민 100%
   - 이는 주민 설득용 표현으로는 강해 보일 수 있으나, 회의록 요약에서는 환각 수치에 해당합니다.

2. **담당자 역할 오류**
   - 원문 역할은 다음과 같습니다.
     - 김민수: 마을 자치 회장
     - 이지아: 이벤트 기획자
     - 박현우: 관리사무소 팀장
   - 그러나 Gemini는 일부 답변에서 김민수를 “관리소장”, 박현우를 “동대표”로 잘못 표기했습니다.

3. **원문에 없는 담당업무 배정**
   - 다음과 같은 업무를 임의로 배정했습니다.
     - 김민수: 배달 존 구역 설정, 주차장 통제 안내문 준비
     - 이지아: 쿠폰 제작
     - 박현우: 수익 기부 세부 절차 협의
   - 원문에서 확실한 담당업무는 아래와 같습니다.
     - 박현우: 금요일까지 광장 전력 점검
     - 이지아: 월요일까지 등불 축제 포스터 시안 제작
     - 김민수: 상가 번영회와 배달 존 협의 마무리
     - 이지아: 주민 참여 메시지 카드 제작
     - 이지아: 실내 동선 준비

4. **마술 공연 확정 표현**
   - 원문에서 마술 공연은 “은퇴 선생님이 가능하다”는 제안 수준입니다.
   - 최종 결정된 메인 프로그램은 야간 등불 축제입니다.
   - Gemini는 일부 답변에서 마술 공연을 확정 프로그램처럼 표현했습니다.

### 수정 결과
- 사용자의 지적 이후 “0%”, “100%” 등 원문에 없는 수치는 제거했습니다.
- 표현력과 주민 친화적 문장은 매우 좋았습니다.
- 다만 역할, 담당업무, 확정 여부를 원문 기준으로 더 엄격히 검증해야 합니다.

## C. GPT-5.5

### 잘한 점
- 수치 검증이 비교적 정확했습니다.
- 원문 수치와 계산 수치를 구분했습니다.
  - 원문 수치: 예산의 40%, 지하 주차장 1구역, 11월 2일 오후 2시
  - 계산 수치: 회의 시간 90분
- 전체 예산 1억 원 기준 렌탈 비용을 4,000만 원으로 정확히 계산했습니다.
- 최종 요약은 간결하고 읽기 쉬웠습니다.

### 문제 발생 지점
1. **확실한 제안내용 누락**
   - GPT-5.5는 “박현우의 확실한 제안내용 없음”이라고 했습니다.
   - 그러나 원문에서 박현우는 여러 제안을 했습니다.
     - 주민 재능 기부 활용
     - 은퇴 선생님 마술 공연 가능성 언급
     - 야간 등불 축제 제안
     - 배달 존 제안
     - 지하 주차장 1구역 활용 제안

2. **김민수 담당업무 누락**
   - GPT-5.5는 “김민수의 확실한 담당업무 없음”이라고 했습니다.
   - 하지만 원문에는 김민수가 “상가 번영회 만나서 배달 존 협의 마무리”를 하겠다고 말했습니다.

3. **먹거리 문제점 누락**
   - “먹거리 문제점 없음”이라고 처리했지만, 원문에는 상권 침해 문제와 항의가 명확히 나옵니다.

4. **사용자 가정을 회의 결정처럼 표현**
   - “전력 충분, 렌탈비 10% 이내일 경우 에어바운스 재검토”는 사용자 가정입니다.
   - 그런데 일부 답변에서 이를 “회의에서 조건부로 다시 판단하기로 했다”처럼 표현했습니다.
   - 실제 회의 원문에는 10% 조건이 없습니다.

5. **메시지 카드 담당자 확인 필요 오류**
   - GPT-5.5는 메시지 카드 담당자를 확인 필요로 처리했습니다.
   - 원문에서는 이지아가 “제가 제작할게요”라고 명확히 말했습니다.
   - 따라서 확인이 필요한 것은 담당자가 아니라 기한입니다.

### 수정 결과
- 수치 관련 오류는 적었습니다.
- 그러나 “확실한 내용만” 추출하는 과정에서 오히려 원문의 확실한 정보를 누락했습니다.
- 최종본은 간결했지만, 담당업무와 문제점 누락을 보완해야 합니다.

# 4. 최종 평가 요약

세 모델 모두 전체 회의 흐름을 파악하는 데는 성공했습니다.  
다만 “확실한 내용만”이라는 조건을 적용했을 때 차이가 뚜렷하게 나타났습니다.

- **Claude Opus 4.8**은 전체적으로 안정적이지만, 일부 확실한 업무와 문제점을 “확인 필요”로 과분류했습니다.
- **Gemini 3.1**은 문장 표현과 주민 친화적 톤은 좋았지만, 원문에 없는 수치와 담당자 역할 오류가 발생했습니다.
- **GPT-5.5**는 수치 검증은 정확했지만, 원문의 확실한 정보인 담당업무와 문제점을 일부 누락했습니다.

# 5. 최종 권장사항

최종 회의록을 만들 때는 다음 원칙을 적용하는 것이 가장 안전합니다.

1. **원문에 있는 수치만 사용하기**
   - 예산의 40%
   - 지하 주차장 1구역
   - 2023년 10월 27일 오후 3:00~4:30
   - 11월 2일 오후 2시
   - 축제까지 약 한 달

2. **사용자 가정은 별도로 표시하기**
   - “광장 전력 용량이 충분하고 렌탈 비용이 전체 예산의 10% 이내일 경우”는 원문 내용이 아니라 추가 검토 조건입니다.

3. **확실한 담당업무는 누락하지 않기**
   - 박현우: 금요일까지 광장 전력 점검
   - 이지아: 월요일까지 등불 축제 포스터 시안 제작
   - 김민수: 상가 번영회와 배달 존 협의 마무리
   - 이지아: 주민 참여 메시지 카드 제작
   - 이지아: 실내 동선 준비

4. **원문 근거가 있는 문제점만 작성하기**
   - 메인행사: 광장 전력 용량 부족, 렌탈 비용이 예산의 40%
   - 먹거리: 외부 푸드트럭으로 인한 상권 침해 문제
   - 쓰레기: 배달 용기 증가 우려
   - 우천 대비: 야외 행사 특성상 날씨 변수, 주차 공간 부족 우려

5. **주민 안내문은 부드럽게 쓰되 사실은 엄격히 유지하기**
   - 친절한 표현은 좋지만, “100%”, “0%”처럼 원문에 없는 수치는 사용하지 않아야 합니다.

# 6. 최종 결론

가장 안정적인 최종 회의록을 만들려면 세 모델의 장점을 조합하는 방식이 적절합니다.

- **Claude의 구조화 능력**
- **Gemini의 주민 친화적 표현력**
- **GPT-5.5의 수치 검증 능력**

이 세 가지를 결합하되, 최종 검수 단계에서는 반드시 원문과 대조하여  
**수치 환각, 담당자 오류, 확실한 정보 누락**을 점검해야 합니다.

따라서 최종 회의록 작성 기준은 다음과 같이 정리할 수 있습니다.

> 원문 수치만 사용하고, 사용자 가정은 별도 표시하며, 확실한 담당업무는 누락하지 않고, 원문 근거가 있는 문제점만 기재한다.

이 기준을 적용하면 주민들에게 신뢰도 높고 이해하기 쉬운 최종 회의록을 제공할 수 있습니다.
